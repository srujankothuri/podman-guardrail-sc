## What is this?

Secure AI Pipeline is an employee-facing HR assistant where every prompt and every response is checked against company compliance rules before anything reaches the user. HR uploads a plain-text compliance document вҖ” that document becomes an active policy layer enforced by [IBM Granite Guardian 3.3 8B](https://huggingface.co/ibm-granite/granite-guardian-3.3-8b) at three independent checkpoints in the pipeline.

The system is intentionally **hybrid**:
- **Local generation** вҖ” answers are generated by Granite 3.3 2B running via Ollama
- **Local guardrails** вҖ” Granite Guardian 3.3 8B runs via Ollama for all compliance checks
- **Containerized runtime** вҖ” all app services run on rootless Podman

### Request flow

```
1. User submits prompt
2. Rate limiter checks per-user and per-IP limits
3. GG1 вҖ” input compliance check (blocks non-compliant prompts immediately)
4. Job enqueued in Redis
5. Worker picks up job
6. Vector DB retrieves relevant knowledge chunks
7. GG2 вҖ” retrieval sufficiency check (retries once if insufficient)
8. Local LLM generates answer from approved context
9. GG3 вҖ” output compliance check (blocks non-compliant responses)
10. Answer delivered to user (with warning if data was insufficient)
```

## Quick start

### 1. Clone the repo

```bash
git clone https://github.com/srujankothuri/podman-guardrail-sc
cd podman-guardrail-sc
```

### 2. Pull models

```bash
# Granite Guardian 3.3 8B вҖ” compliance guardrail
ollama pull hf.co/ibm-granite/granite-guardian-3.3-8b-GGUF

# Granite 3.3 2B вҖ” answer generation
ollama pull granite3.3:2b
```

> First pull downloads ~4.5GB and ~1.5GB respectively. Subsequent starts are instant.

### 3. Set up environment

```bash
cp .env.example .env
```

### 4. Start backend

```bash
cd backend
python3 -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt

# Start Redis (if not already running)
brew services start redis       # Mac
# or: redis-server

# Start API server
uvicorn app.main:app --reload
```

### 5. Ingest sample knowledge

```bash
# In a new terminal, inside backend/ with venv activated
python3 ingest_sample.py
```

### 6. Start frontend

```bash
cd frontend
npm install
npm run dev
```

### 7. Open the app

| URL | Description |
|---|---|
| `http://localhost:5173/chat` | Employee chat interface |
| `http://localhost:5173/admin` | HR admin вҖ” policy upload + audit log |
| `http://localhost:8000/docs` | Backend Swagger UI |
| `http://localhost:8000/health` | Health check |

## Run with Podman

```bash
# Make sure Podman is running rootless
podman machine start
podman system connection default podman-machine-default-rootless

# Build and start all services
podman compose up --build

# Ingest sample knowledge into the running container
podman exec secure-ai-pipeline-backend-1 python3 ingest_sample.py
```

> The app uses Docker-compatible images and Compose syntax, but runs on the Podman engine in rootless mode.

## API reference

### `POST /api/prompts`
Submit a user prompt. Runs GG1 check before queuing.

```json
// Request
{ "prompt": "What is our leave policy?", "user_id": "employee-1" }

// Response 200
{ "job_id": "abc123", "status": "queued", "message": "Your request has been queued." }

// Response 400 вҖ” GG1 blocked
{ "detail": "Your prompt could not be processed under company compliance rules. Please rephrase and try again." }

// Response 429 вҖ” Rate limited
{ "detail": "You are sending requests too quickly. Please wait a moment." }
```

### `GET /api/jobs/{job_id}`
Poll job status. Triggers worker on first poll.

```json
// Processing
{ "job_id": "abc123", "status": "retrieving_knowledge", "insufficient_data": false }

// Completed
{ "job_id": "abc123", "status": "completed", "answer": "...", "insufficient_data": false }

// GG3 blocked
{ "job_id": "abc123", "status": "blocked", "message": "A response could not be generated..." }
```

### `POST /api/admin/policy/upload`
Upload a `.txt` compliance document. Parses and activates immediately.

### `GET /api/admin/policy/current`
Returns active policy version, rule count, and hash.

### `GET /api/admin/audit/events`
Returns last 50 audit events in reverse chronological order.

## Compliance policy format

HR uploads a plain `.txt` file with numbered rules:

```
1. Do not disclose employee salaries or compensation bands.
2. Do not provide legal advice beyond approved policy summaries.
3. Do not share personally identifiable information.
```

The backend parses this into a versioned JSON policy with auto-detected severity, category, and action for each rule. Previous versions are archived automatically.

## Audit log events

Every pipeline stage emits a structured JSONL event to `backend/logs/YYYY-MM-DD.jsonl`:

| Event | Meaning |
|---|---|
| `gg1_allowed` | Input passed compliance check |
| `gg1_blocked` | Input blocked вҖ” user gets nudge message |
| `gg1_error` | Guardian timed out вҖ” logged as technical failure |
| `gg2_sufficient` | Retrieved context is relevant and complete |
| `gg2_insufficient` | Context insufficient вҖ” retry triggered |
| `llm_started` | LLM generation began |
| `llm_completed` | LLM returned answer |
| `gg3_allowed` | Output passed compliance check |
| `gg3_blocked` | Output blocked вҖ” security alert created |
| `security_alert_created` | Triggered on GG3 block for review team |
| `response_delivered` | Answer successfully shown to user |
| `rate_limit_blocked` | Request rejected by rate limiter |

## Hackathon requirement mapping

| Requirement | Implementation |
|---|---|
| Rootless containers | Podman engine, rootless default connection |
| Local AI | Granite 3.3 2B + Guardian 3.3 8B via Ollama |
| Input/output guardrails | GG1 (input) + GG3 (output) via Granite Guardian |
| Content filtering | Policy-based rejection at GG1 and GG3 |
| Audit logging | Structured JSONL events for every pipeline stage |
| Rate limiting | Redis sliding window, before queue |
| Graceful adversarial handling | Friendly nudge messages, never exposes policy details |
| Network isolation | Public net (frontend + backend) / internal net (Redis, worker) |
